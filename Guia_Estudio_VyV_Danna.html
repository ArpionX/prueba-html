1. Fundamentos de Verificación y Validación

El testing dinámico se ejecuta con el código corriendo, a diferencia del análisis estático. Aquí están las preguntas del millón para tu examen:

Verificación (¿Cómo?)

Evalúa si el software cumple con los requisitos técnicos de diseño. ¿Estamos construyendo el producto correctamente?

Cumplir especificación

Validación (¿Qué?)

Comprueba si el software cumple las expectativas reales del cliente y usuario final. ¿Estamos construyendo el producto correcto?

Satisfacer al cliente

Diferencia clave con Testing Estático

El Testing Estático se realiza mediante inspecciones, revisiones de código y análisis sin ejecutar el sistema (como buscar variables no usadas). El Testing Dinámico requiere obligatoriamente que el software se ejecute para comprobar comportamientos.

2. Técnicas de Caja Negra

Se enfocan puramente en las entradas y salidas, sin mirar el código interno.

2.1 Partición de Equivalencia

Consiste en dividir los datos de entrada en grupos (clases) que se procesan de la misma forma. Solo necesitamos probar un representante de cada grupo.

Caso de Estudio real (V&V.pdf): Una función validarEdad(int edad) que acepta edades entre 18 y 60 años inclusive.
  • Clase Válida: 18 a 60 (Representante: 25)
  • Clase Inválida Inferior: menor a 18 (Representante: 15)
  • Clase Inválida Superior: mayor a 60 (Representante: 65)

2.2 Análisis de Valores Límites

Los programadores solemos equivocarnos en los bordes (confundir < con <=). Por eso probamos los límites exactos:

Punto Crítico Valores a Probar Resultado Esperado
Límite Inferior (18)17 (Justo abajo), 18 (Justo en el límite), 19 (Justo arriba)17: Inválido | 18, 19: Válido
Límite Superior (60)59 (Justo abajo), 60 (Justo en el límite), 61 (Justo arriba)59, 60: Válido | 61: Inválido

3. Técnicas de Caja Blanca

Aquí sí abrimos el código y evaluamos su flujo interno.

3.1 Criterios de Cubrimiento (Coverages)

Los tres niveles principales que debes saber para estructurar tus casos de prueba:

1. Cubrimiento de Sentencias/Comandos

Garantiza que cada línea de instrucción de código se ejecute al menos una vez.

2. Cubrimiento de Decisiones o Ramas

Asegura que cada bifurcación (True / False de cada if) se evalúe por completo.

3. Cubrimiento de Condiciones

Evalúa de forma independiente cada condición lógica simple dentro de un condicional compuesto (ej: edad >= 18 && peso >= 50).

3.2 Complejidad Ciclomática (Métrica Esencial)

Es una métrica matemática de software que indica el número de caminos linealmente independientes en un código.

Fórmula de McCabe:
V(G) = E - N + 2
Donde:
• E = Número de aristas/flechas (edges) en el grafo de flujo.
• N = Número de nodos/cajas (nodes) en el grafo de flujo.
